# Claude Code Instructions ‚Äî fivekmrun_app_flutter

## Project Overview
Flutter mobile app for [5kmRun.bg](https://5kmrun.bg). Targets Android and iOS. The app authenticates users via username/password or user ID, displays run history, charts, barcode, settings, and integrates with Strava.

## Repository Layout
- `lib/` ‚Äî all Dart source code, feature-based folders (`login/`, `home/`, `barcode/`, `offline_chart/`, `state/`, `l10n/`, etc.)
- `test/common/` ‚Äî unit and widget tests; `flutter test` is run in CI on every PR
- `android/` / `ios/` ‚Äî native platform projects
- `../.github/workflows/` ‚Äî CI lives one level up at the repo root, not inside this Flutter project

## Secrets & Private Files
`lib/private/secrets.dart` is generated by CI (never committed). It exports `stravaSecret`, `stravaClientId`, `googleMapsKey`. If it is missing locally, create it manually with placeholder values so the project compiles.

## CI Pipeline
- Triggered on every PR to `master`
- Builds Android (`flutter build appbundle`) and iOS (`flutter build ios --no-codesign`)
- Runs `flutter test` on both platforms
- Flutter version is controlled by the `FLUTTER_VERSION` repo variable ‚Äî check this before suggesting version upgrades

## Workflow Preferences
- The user gives high-level goals and expects autonomous execution ‚Äî minimal check-ins unless blocked
- Prefer targeted file edits (`Edit` tool) over shell-based file manipulation ‚Äî it's faster and more reviewable
- Read relevant files before proposing or making changes
- For non-trivial tasks, state your plan in 2‚Äì3 bullet points before starting

## Branching & PRs
- Branch naming: `claude-fix/<short-description>` or `claude-feat/<short-description>`
- Always push the branch and open a PR ‚Äî the user reviews before merging
- The git remote `origin` points to `https://github.com/5kmrun-bg/fivekmrun-app.git` (repo was transferred from `etabakov/fivekmrun-app`)

## Testing
- All tests live under `test/` and are run with `flutter test`
- Prefer **widget tests** over integration tests: they run on the host (no emulator), are fast, and work in CI out of the box
- When fixing a bug, add tests that cover the broken cases before/alongside the fix
- Integration tests requiring a device/emulator are not currently wired into CI ‚Äî do not add them without discussion

## Code Conventions
- Use `int.tryParse()` instead of `int.parse()` ‚Äî never let user input cause an unhandled exception
- Add `mounted` checks before `setState` calls inside async callbacks to prevent setState-after-dispose crashes
- Prefer `launchUrl()` (url_launcher 6.1+) over the deprecated `launch()`
- Localisation strings live in `lib/l10n/app_bg.arb` and `lib/l10n/app_en.arb` ‚Äî keep both in sync when adding new strings

## Preparing a New Release (`/release`)

When the user asks to prepare a new release or invokes `/release`, follow these steps:

1. **Determine version bump** ‚Äî If the user did not specify major/minor/patch, inspect the commits since the last tag (`git log <last-tag>..HEAD --oneline`) and ask: "Should this be a minor or patch bump?" (ask about major only if there are breaking changes). Wait for the answer before proceeding.

2. **Bump the version in `pubspec.yaml`** ‚Äî Current format is `MAJOR.MINOR.PATCH+BUILD`. Increment the appropriate semver segment and increment the build number by 1. Edit `pubspec.yaml` directly.

3. **Commit and push the version bump** ‚Äî Commit only `pubspec.yaml` with message `chore: bump version to X.Y.Z+N`, then push directly to `master`.

4. **Collect the changelog** ‚Äî Run `git log <last-tag>..HEAD --pretty=format:"- %s"` to list all commits since the previous tag. Clean up the list (remove merge commits, CI noise). This becomes the tag description.

5. **Create and push the release tag** ‚Äî Create an annotated tag named `vX.Y.Z` on `master` with the changelog as the tag message, then push it:
   ```
   git tag -a vX.Y.Z -m "$(changelog)"
   git push origin vX.Y.Z
   ```

6. **Generate store release notes** ‚Äî Translate the changelog into user-facing language (no technical jargon, bullet points, max 500 characters each). Write two files:
   - `whatsnew/whatsnew-bg` ‚Äî Bulgarian
   - `whatsnew/whatsnew-en-GB` ‚Äî English

   Format (plain text, no headings, bullet character `‚Ä¢`):
   ```
   ‚Ä¢ User-facing change 1
   ‚Ä¢ User-facing change 2
   ```

   Commit both files directly to `master` with message `chore: add release notes for vX.Y.Z` and push.

7. **Trigger the deployment workflows** ‚Äî Both workflows use `workflow_dispatch`. Trigger them via the GitHub CLI:
   ```
   gh workflow run upload-appstore.yml --repo 5kmrun-bg/fivekmrun-app
   gh workflow run upload-playstore.yml --repo 5kmrun-bg/fivekmrun-app
   ```
   Report the run URLs so the user can monitor progress.
